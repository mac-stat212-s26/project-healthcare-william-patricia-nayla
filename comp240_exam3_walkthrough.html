Q1: Does check have good, moderate, or poor spatial locality?
void check(int *A, int len) { for (int i = 0; i < len; i++) { if (A[i] < 0) { return 1; } } return 0; }
GOOD spatial locality
Why: The loop accesses A[0], A[1], A[2]... in sequential order. When the CPU loads A[0] from memory, it also loads the next several integers into the same cache line. Every subsequent access hits the cache. This is the best-case pattern.
Key insight: Sequential forward iteration = maximum cache hit rate.
Q2: Does check2 (backward loop) have good, moderate, or poor spatial locality?
void check2(int *A, int len) { for (int i = len-1; i >= 0; i--) { if (A[i] < 0) { return 1; } } return 0; }
GOOD spatial locality
Why: Even though we go backward, when the cache loads a chunk of memory it loads the surrounding elements. Going from A[len-1] to A[len-2] is still sequential within a cache line — those elements are all in the same block.
Key insight: The cache loads a whole block at once. Backward iteration still accesses elements within the same block consecutively.
Q3: Does sum_ends have good, moderate, or poor spatial locality?
void sum_ends(int *A, int len) { for (int i = 0; i < len/2; i++) { A[i] = A[i] + A[len - i - 1]; } }
GOOD / MODERATE spatial locality
Why: Two references per loop pass — A[i] moves forward from the front, A[len-i-1] moves backward from the end. Both are accessing contiguous values, just from opposite ends.
Tradeoff: More misses than a pure forward loop (two cache lines being used simultaneously), but both references are still sequential — not random. So better than moderate, not quite as good as a single sequential access.
Q4: Does sumRows have good, moderate, or poor spatial locality?
/* B is statically declared 2D array with N rows, M cols */ void sumRows(int B[N][M], long res[N]) { for (int r = 0; r < N; r++) { row_tot = 0; for (int c = 0; c < M; c++) { row_tot += B[r][c]; // inner loop moves by column } res[r] = row_tot; } }
GOOD spatial locality
Why: In C, 2D arrays are stored in row-major order: B[0][0], B[0][1], B[0][2]... B[0][M-1], B[1][0], B[1][1]... The inner loop walks across columns (c++) which means consecutive memory addresses. This is the ideal access pattern for a 2D array.
Rule: Row-major access (inner loop varies column) = GOOD for C arrays.
Q5: Does sumCols have good, moderate, or poor spatial locality?
void sumCols(int B[N][M], long res[M]) { for (int c = 0; c < M; c++) { row_tot = 0; for (int r = 0; r < N; r++) { row_tot += B[r][c]; // inner loop moves by row! } res[c] = row_tot; } }
POOR spatial locality
Why: The inner loop increments r, so we access B[0][c], B[1][c], B[2][c]... Each step skips an entire row (M integers) in memory. With large M, each access is in a completely different cache block — very high miss rate.
Rule: Column-major access (inner loop varies row) = POOR for C arrays. Every access jumps M×4 bytes forward in memory.
Q6: Does shuff have good, moderate, or poor spatial locality?
void shuff(int C[N], int D[N]) { int j; for (int i = 0; i < N; i++) { j = rand() % N; D[i] = C[j]; // D accessed sequentially, C randomly } }
POOR / MODERATE spatial locality
D[i]: Sequential — good spatial locality on D's side.
C[j]: Random index — no spatial locality. For large arrays, C[rand()%N] is almost always a cache miss.
Overall: The random accesses to C dominate and give poor locality overall. The sequential D access helps a little, so some sources say poor/moderate.
Q7: For time1, which variables have GOOD vs POOR temporal locality?
long time1(int x, int y, int z) { long running_sum = x; long final_sum = x + y + z; for (int i = x; i < z; i++) { if (x > y) { running_sum += y; } else { running_sum += x; } } if (running_sum > 0) { final_sum = running_sum; } return final_sum; }
GOOD temporal locality: i, x, y, z, running_sum
These are used every single iteration of the loop — they stay "hot" in registers/cache.
POOR temporal locality: final_sum
final_sum is set once at the beginning (x+y+z), then only conditionally updated once at the end. It is NOT reused inside the loop at all — long gap between uses = poor temporal locality.
Q8: For time2, which variables have GOOD vs POOR temporal locality?
int time2(int *A, int len) { int first = A[0]; int last = A[len-1]; int max = A[1]; for (int i = 1; i < len-1; i++) { if (A[i] > max) { max = A[i]; } } return first + max + last; }
GOOD temporal locality: A, len, max, i
A is accessed every iteration, len is used in the loop condition each time, max and i are checked/updated each iteration.
POOR temporal locality: first, last
Both are set once at the top, then sit unused through the entire loop, only referenced at the final return. Long gap between first use and last use = poor temporal locality.

J1: Direct-mapped cache, 64 KB total, block size 16 bytes. How many lines?
Step 1: Convert 64 KB to bytes: 64 × 1024 = 65,536 bytes
Step 2: Lines = total bytes ÷ block size = 65,536 ÷ 16 = 4,096 lines
Formula: # lines = cache_size_bytes / block_size_bytes
J2: Two-way set-associative, 1024 sets, block size 32 bytes. Total storage?
Formula: total = sets × ways × block_size
= 1024 sets × 2 lines/set × 32 bytes/line
= 1024 × 64 = 65,536 bytes = 64 KB
Note: 2^10 sets × 2^6 bytes/set = 2^16 bytes = 64 KB

J4: M=128 rows, N=64 cols int array. Direct-mapped, 32 KB, block=32 bytes. Answer the 5 sub-questions.
Questions: (a) How many lines does the cache have? (b) How many integers can fit in one line? (c) How many lines to store one row of the array? (d) How many rows can fit in cache at once? (e) What if we change N to 128?
(a) 32 KB = 32,768 bytes. Lines = 32,768 / 32 = 1,024 lines
(b) 32 bytes / 4 bytes per int = 8 integers per line
(c) One row has 64 ints. 64 ints / 8 ints per line = 8 lines per row
(d) 1,024 lines / 8 lines per row = 128 rows — the entire array fits! No conflict misses.
(e) With N=128: each row needs 128/8 = 16 lines. 1,024/16 = 64 rows fit. With 128 rows total, only half fit → conflict misses possible.
J5: Miss rate for initialize2 with #define N 64, same 32 KB / 32-byte cache. Then what if line size doubles to 64 bytes?
void initialize2(int array[M][N]) { for (i = 0; i < M; i++) { for (j = 0; j < N; j++) { array[i][j] = i * j; } } }
With N=64, 32-byte line: Each line holds 32/4 = 8 ints. For every 8 consecutive j values, there is 1 miss (first access) then 7 hits.
Miss rate = 1/8 = 12.5%
Why no conflict misses? The entire 128×64 array (32,768 bytes = 1,024 lines) fits perfectly in the 1,024-line cache. Each block is loaded once and never evicted.
With 64-byte line: Now 64/4 = 16 ints per line. Miss rate = 1/16 = 6.25%
J6: copy_matrix with ROWS=64, COLS=128. Cache 64 KB, block 16 bytes. What is the miss rate? What happens at ROWS=128?
/* src starts at address 0, dst starts right after src */ void copy_matrix(int dst[ROWS][COLS], int src[ROWS][COLS]) { for (i = 0; i < ROWS; i++) for (j = 0; j < COLS; j++) dst[i][j] = src[i][j]; }
Cache: 64 KB = 65,536 bytes → 65,536/16 = 4,096 lines
src size: 64×128×4 = 32,768 bytes → 2,048 lines (lines 0–2047)
dst size: 32,768 bytes → lines 2048–4095. No overlap with src! They both fit without conflict.
Miss rate (ROWS=64): Each line holds 16/4=4 ints. Inner loop: 1 miss per 4 j-values for src + 1 miss per 4 for dst = 2 misses, 6 hits per 4 iterations → 2/8 = 25%
ROWS=128: Now src = 65,536 bytes → 4,096 lines (0–4095). dst also 4,096 lines but maps to lines 0–4095 (same!). Every dst[i][j] access evicts the corresponding src block → 100% miss rate (thrashing)

K1: 14-bit addresses, 2-way set-associative, block=4 bytes, 16 sets. How many bits for offset, index, tag?
Offset bits = log₂(block_size) = log₂(4) = 2 bits (to index 4 bytes within a block)
Index bits = log₂(num_sets) = log₂(16) = 4 bits (to select 1 of 16 sets)
Tag bits = address_bits − offset − index = 14 − 2 − 4 = 8 bits
Format (MSB→LSB):
TTTTTTTTIIIIOO
Red = tag, Blue = index, Green = offset

Exam Q9a: Does shuffle have good or poor spatial locality?
#define N 30000 void shuffle(int array[N]) { int r1, r2; for (int i = 0; i < N/2; i++) { r1 = (int)rand() % N; r2 = (int)rand() % N; int t = array[r1]; array[r1] = array[r2]; array[r2] = t; } }
POOR spatial locality
r1 and r2 are random indices → array[r1] and array[r2] are random memory addresses. With N=30,000, the probability that two consecutive accesses hit the same cache line is near zero. Almost every array access is a miss.
Exam Q9b: Does init_array1 have good or poor spatial locality?
#define N 20000 void init_array1(int array[n], int i) { for (int i = 0; i < N; i++) { array[i] = i * i; } }
GOOD spatial locality
Sequential forward access pattern: array[0], array[1], array[2]... Each cache block load gives multiple hits on the subsequent elements. Optimal for cache performance.
Exam Q9c: Which variables in avg_mat have GOOD temporal locality?
#define M 2000 #define N 5000 float avg_mat(int A[M][N]) { int i, j; float size = M * N; float sum = 0; for (i = 0; i < M; i++) { for (j = 0; j < N; j++) { sum += A[i][j]; } } return sum / size; }
GOOD temporal locality: i, j, sum
i — used and updated every outer loop iteration. j — used every inner loop iteration. sum — updated on every single iteration of the inner loop (most frequently used).
MODERATE/POOR: A (pointer accessed every step but elements not reused), size (computed once, used only at return statement)
Exam Q10a-b: short A[256][64]. Cache: 512 lines, direct-mapped, 32-byte blocks. (a) Lines per row? (b) Rows in cache?
#define M 256 #define N 64 void manipulate(short A[M][N]) { int half = M/2; // = 128 for (i = 0; i < half; i++) for (j = 0; j < N; j++) A[i][j] += A[i+half][j]; }
(a) short = 2 bytes. Row size = 64 × 2 = 128 bytes. Lines per row = 128 / 32 = 4 lines per row
(b) 512 lines / 4 lines per row = 128 rows fit in cache
Exam Q10c: What is the miss rate for manipulate?
A starts at memory location 0. Row i maps to cache lines (i×4) mod 512. Row i+128 maps to ((i+128)×4) mod 512 = (i×4+512) mod 512 = (i×4) mod 512
~100% miss rate (thrashing)
The conflict: Row i and row i+128 (which is i+half) map to the EXACT same 4 cache lines (since 128×4 = 512 ≡ 0 mod 512).
The inner loop body: A[i][j] += A[i+half][j]
→ Read A[i][j]: if not in cache (evicted by previous A[i+half] access) → MISS, load row i's block
→ Read A[i+half][j]: conflicts with row i's block → MISS, evicts row i
→ Write A[i][j]: row i was just evicted → MISS again!
Every access causes a conflict miss. Miss rate ≈ 100%
Exam Q10d: Move to a machine with 1024-line cache, 32-byte blocks. New miss rate?
Now: Row i maps to lines (i×4) mod 1024. Row i+128 maps to (i×4+512) mod 1024 = (i×4+512). These are 512 lines apart → NO CONFLICT.
Each row now has its own section of cache. Accesses are row-major (inner loop moves by column j).
Shorts per line = 32/2 = 16. So 1 miss per 16 j-values per row. Two rows accessed per i: 2 misses per 16 shorts × 2 = 2 misses / 32 accesses.
Miss rate = 1/16 = 6.25% (compulsory misses only)
